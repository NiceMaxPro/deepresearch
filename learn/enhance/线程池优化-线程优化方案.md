# GraphProcess 线程池优化方案

## 现状分析

`GraphProcess.java` 中使用共享线程池执行图流处理任务：

```java
// 第 73 行
private final ExecutorService executor = Executors.newFixedThreadPool(10);
```

```java
// 第 121 行
Future<?> future = executor.submit(() -> {
    generator.doOnNext(...).doOnComplete(...).doOnError(...).subscribe();
});
```

每个任务的核心工作是调用 `Flux.subscribe()` 阻塞等待图执行完成，单个任务可能持续数分钟（LLM 调用、搜索引擎请求等）。线程在此期间完全处于阻塞等待状态，不执行任何计算工作。

## 三个问题

### 问题一：线程耗尽 + 无界队列

`Executors.newFixedThreadPool(10)` 内部使用 `LinkedBlockingQueue`（无界队列）。10 个线程全部被长时间任务占用时，第 11 个请求进入内存队列无限等待，而不是被拒绝。

```
线程 1-10: [████████ 研究中...]（每个持续数分钟）
队列:    [请求11] [请求12] [请求13] ... [请求N]

后果：前端 HTTP 连接建立但无数据返回，用户看到浏览器无限转圈；
     队列无限增长，极端情况下导致 OOM。
```

### 问题二：sessionCountMap 内存泄漏

```java
private final ConcurrentHashMap<String, Integer> sessionCountMap = new ConcurrentHashMap<>();

public GraphId createNewGraphId(String sessionId) {
    int count = sessionCountMap.merge(sessionId, 1, Integer::sum);
    // ↑ 只增不减，session 过期后 key 永远残留
}
```

每次请求向 Map 新增或递增一个 key，没有过期机制、没有上限、没有清理逻辑。JVM 重启前持续增长。

### 问题三：队列中的任务无法取消

```java
Future<?> future = executor.submit(...);   // 返回 Future
graphTaskFutureMap.put(graphId, future);   // 存入 Map 供取消

public boolean stopGraph(GraphId graphId) {
    Future<?> future = graphTaskFutureMap.remove(graphId);
    if (future == null) return false;      // 任务还在队列里排队时，这里拿到的是 null
    return future.cancel(true);
}
```

任务提交到线程池后：如果线程池有空闲线程，任务立即执行，`submit()` 返回 Future 存入 Map；如果线程池已满，任务进入队列等待，此时虽然 `submit()` 返回了 Future，但该 Future 代表的是一个**尚未开始执行**的任务。如果此时用户调用 stopGraph，`graphTaskFutureMap.remove(graphId)` 可以拿到这个 Future，`future.cancel(true)` 也能取消队列中的任务——这一点我之前说错了。但问题是，这需要前端主动调用 `/chat/stop`，而大多数用户不会这么做。实际效果是：用户看着转圈关掉页面，任务继续蹲在队列里。

## 优化方案：方法内创建线程，用完即销毁

### 核心思路

去掉共享线程池。每次请求 new 一个专用线程，任务结束线程自动死亡。用 `AtomicInteger` 控制并发上限，满了直接返回错误给前端。

### 改造后的 processStream

```java
// 删掉
// private final ExecutorService executor = Executors.newFixedThreadPool(10);

// 新增并发计数器
private final AtomicInteger activeThreads = new AtomicInteger(0);
private static final int MAX_CONCURRENT = 50;

public void processStream(GraphId graphId, Flux<NodeOutput> generator,
                          Sinks.Many<ServerSentEvent<String>> sink) {
    // 并发上限检查
    if (activeThreads.incrementAndGet() > MAX_CONCURRENT) {
        activeThreads.decrementAndGet();
        sink.tryEmitNext(ServerSentEvent.builder(
            "{\"error\":\"服务器繁忙，请稍后重试\"}").build());
        sink.tryEmitComplete();
        return;
    }

    final String graphIdStr = safeObjectToJson(graphId);

    Thread worker = new Thread(() -> {
        try {
            generator
                .doOnNext(output -> {
                    String nodeName = output.node();
                    String content;
                    if (output instanceof StreamingOutput streamingOutput) {
                        content = buildLLMNodeContent(nodeName, graphId, streamingOutput, output);
                    } else {
                        content = buildNormalNodeContent(graphId, nodeName, output);
                    }
                    if (StringUtils.isNotEmpty(content)) {
                        sink.tryEmitNext(ServerSentEvent.builder(content).build());
                    }
                })
                .doOnComplete(() -> {
                    logger.info("Stream processing completed.");
                    sink.tryEmitComplete();
                })
                .doOnError(e -> {
                    logger.error("Error in stream processing", e);
                    sink.tryEmitNext(ServerSentEvent.builder(
                        String.format(TASK_STOPPED_MESSAGE_TEMPLATE, graphIdStr, "服务异常")).build());
                    sink.tryEmitError(e);
                })
                .subscribe();
        } catch (Throwable t) {
            // 兜底：subscribe 之前炸了，或 doOnError 自身炸了
            logger.error("Worker thread fatal error", t);
            sink.tryEmitNext(ServerSentEvent.builder(
                String.format(TASK_STOPPED_MESSAGE_TEMPLATE, graphIdStr, "服务内部错误")).build());
            sink.tryEmitError(t);
        } finally {
            activeThreads.decrementAndGet();
            graphTaskFutureMap.remove(graphId);
        }
    }, "deepresearch-" + graphId.threadId());
    worker.setDaemon(true);
    worker.start();

    graphTaskFutureMap.put(graphId, worker);
}
```

### 改造后的 stopGraph

```java
public boolean stopGraph(GraphId graphId) {
    Thread worker = graphTaskFutureMap.remove(graphId);
    if (worker == null) {
        return false;
    }
    if (!worker.isAlive()) {
        return true;
    }
    worker.interrupt();
    return true;
}
```

### 同步修改 createNewGraphId

```java
public GraphId createNewGraphId(String sessionId) {
    if (StringUtils.isEmpty(sessionId)) {
        throw new IllegalArgumentException("Session Id is empty");
    }
    return new GraphId(sessionId,
        sessionId + "-" + System.currentTimeMillis());
}
```

用时间戳替代 HashMap 计数器，消除 `sessionCountMap` 内存泄漏。

## 改造前后对比

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| 线程管理 | 共享线程池 10 线程 | 每次 new Thread，用完即销毁 |
| 排队行为 | 无界队列，无限等待 | 无队列，并发满即返回错误 |
| 取消机制 | Future.cancel(true) | Thread.interrupt()，直接打断 |
| 超限响应 | 浏览器永远转圈 | 立即返回错误 JSON |
| Map 清理 | doOnComplete 清，异常路径遗漏 | finally 保证必清 |
| sink 关闭 | 异常时不一定 | catch 块保证必关 |
| 异常可见性 | 被 Future/WrapperRunnable 吞掉 | catch 块日志可见 |
| sessionCount | HashMap 只增不减 | 时间戳替代，无泄漏 |
| 线程开销 | 10 个持久线程 | 每请求一个短命线程，通常 1MB 栈/个 |

## 打断机制说明

`Thread.interrupt()` 打到 `subscribe()` 上时：

1. `subscribe()` 内部通过 `CountDownLatch.await()` 或 `LockSupport.park()` 阻塞
2. `interrupt()` 触发 `InterruptedException`
3. Reactor 包装为 `RuntimeException`
4. 触发 `doOnError` 回调 → sink 收尾 → 线程退出

与 `Future.cancel(true)` 效果完全一致，因为后者底层也是调 `Thread.interrupt()`。

## 代价分析

| 代价 | 说明 |
|------|------|
| 线程创建开销 | 每次 new Thread 约几微秒，相比数分钟的任务执行可以忽略 |
| 线程栈内存 | 每线程约 1MB，50 并发 = 50MB，现代服务器完全可接受 |
| CPU 消耗 | 线程大部分时间在 I/O 阻塞等待（LLM API、搜索引擎），不消耗 CPU |
| 线程切换 | 50 个阻塞线程的上下文切换开销极低，远低于 10 线程池 + 大量排队任务的内存压力 |

真正的瓶颈是 LLM API 的并发限制和调用配额，而非服务端线程数。
